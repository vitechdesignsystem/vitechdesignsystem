Buttons

Buttons are clickable or tappable items used to perform an action or initiate a process. Our buttons come in three major sizes (default, small and large) and three types (primary, secondary, and tertiary). When you design a page with multiple buttons, you need to prioritize the actions and choose the right type to highlight the main actions.

Primary buttons have the highest degree of emphasis so they are used for the action most users should click or tap. There should be only one primary button on a page at a time. Other buttons should be displayed as secondary buttons or tertiary buttons. Not all screens require a primary button. Please refer to the Use Cases section of this page for our recommendation on how buttons are to be used.

With Icon

Buttons with icons are recommended when you need more visual attention among content and other types of buttons. When you choose an icon, its visual representation must be relatable to the action. Avoid the use of icons for merely aesthetic purposes. Never use more than a single icon within a button.

In Widgets

This section describes how buttons must be used within widgets.

Actions on Widget Headers

Widgets can contain up to two actions as tertiary buttons which are located on the header area. These actions should impact the entire widget (like adding a new record etc.). It is recommended to prioritize the top two actions that the users are most likely to perform. The rest are located inside the "More" stacked button.

Recommended
  • Use only upto max of 2 actions on the widget header.
  • Only use tertiary buttons for widget level actions.
Don't
  • Don’t use primary or secondary buttons on the widget header.
Lists and Grids

Lists and grid can optionally contain buttons to provide quick actions. On lists, this is done as a separate row that houses the actions. On data grids/tables, the first column is reserved for actions. Note that on data tables, the overflow action is located first.

These actions only affect the individual record and do not control any other records in the widget, or cannot impact any other elements on the page.

Recommended
  • Lists can have a max of 2 actions as tertiary buttons. The rest are tucked inside the overflow menu.
Recommended
  • Use upto two tertiary buttons for grid actions. The rest must be tucked inside the overflow menu.
Filters and Groups

As with actions on lists and data tables, whenever there are more than two widget actions, they are located inside the "More" stacked button. Filters provide a convenient way to sort, search and filter the data present inside the widget. In some cases, they allow the user to toggle the visible columns in a data table.

Button groups are used to aggregate data into different groups. This allows the data to be presented an organized fashion, while letting users toggle between the groups.

Recommended
  • Only a max of two stacked buttons can be present on the widget header (for “More” and “Filter” actions).
Recommended
  • Use button groups within widgets to toggle between sets of data.

Although the Experience Designer allows changing of button styles, we strongly discourage using primary and secondary buttons on widget headers and within the widgets. Button types (both size and style) follow a certain hierarchy that is appropriate only at the recommended locations.

Using incorrect button styles would distract users from wayfinding and performing tasks.

Don't
  • Don't combine different button types at inappropriate places. These compete for attention with each other and the hierarchy of the page is confusing.

In Modals

Buttons are used on modals to perform actions and for navigation. The bottom of the modal houses a fixed button bar that generally contains positive CTA button and a cancel. Ocassionally, there might also be a secondary action next to the positive CTA.

On modals and wizards, there must be only one primary CTA button to improve wayfinding and inform the users about the next steps in the process. The primary CTA must be disabled and enabled according to the fields/conditions of the modal. This is a way to tacitly inform the user if there is any required field that hasn't been filled yet, or filled inapropriately.

The buttons on a wizard are limited to Cancel, Back and Next/Confirm (with the next/confirm being the CTA). When a user has to fill a form on the wizard step or make a requried selection, etc., the CTA is disabled. This allows the user to quickly understand when they can proceed to the next screen.

Recommended
  • Enable and disable the button states appropriately, when there is insufficient or incorrect information.
Don't
  • Don't use more than two actions on the modal's bottom right button bar area.
